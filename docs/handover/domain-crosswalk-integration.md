# Plan: Integrate 419 Deep Domain Taxonomies Into the Crosswalk Graph

> **Status: Completed (2026-04).** All 434 curated domain taxonomies are bridged to NAICS/ISIC/NACE and `edge_kind` labeling is live on every equivalence response. This document is retained for historical context on the sector-anchor generator design.

## Context

WorldOfTaxonomy has 434 domain systems (IDs starting with `domain_`). Of those:

- **15** are crosswalk-integrated via the five pilot ingesters (`crosswalk_naics11_domains.py`, `naics21`, `naics22`, `naics23`, `naics484`). These cover the original truck / agriculture / mining / construction / utilities pilots.
- **419** are **islands** - zero `equivalence` edges in either direction.

Consequence in the live product: classify a query like "telemedicine platform" and the result surfaces `dhlcit_ehr_ambulatory` (a deep-domain vocabulary) as a top match with no bridge to the NAICS / ISIC / NACE code the user's accountant needs. The two-category UI already tells the user "Start here: Domain taxonomies -> Official standard codes", but for 96.5% of deep domains the promised bridge doesn't exist in the data.

A second gap surfaces once the bridges exist: every crosswalk response mixes three kinds of edges (standard<->standard, standard<->domain, domain<->domain) with no label telling the user or agent which is which. Consumers re-derive it by eyeballing system-ID prefixes.

**Goals:**

1. Wire every deep-domain taxonomy to at least one NAICS 2022 anchor with `match_type='broad'` so every domain match has a bridge to a standard reporting code.
2. Label every equivalence in every API/MCP response with an `edge_kind` so consumers can filter cleanly without re-deriving it.

Both goals land together across a single set of phases so the documentation, tests, and code touch each file exactly once.

## Recommended approach

Replicate the existing pilot pattern (`world_of_taxonomy/ingest/crosswalk_naics484_domains.py`) at scale, but generate the `(naics_code, domain_system, domain_root_code)` tuples programmatically from a single mapping table instead of hand-crafting 419 modules. Add a pure-function `compute_edge_kind()` in `category.py` so every new edge (and every legacy edge) picks up a label with no schema change.

Three layers:

1. **Anchor table** - one row per domain system, maps `domain_system_id` to one or more NAICS 2022 anchor codes (usually 2-3 digit). Lives as a JSON file.
2. **Edge generator + edge-kind labeling** - walks the anchor table, resolves domain top-level codes, emits `equivalence` rows. `compute_edge_kind()` lights up labels on both new and legacy edges via the response serializer.
3. **Transitive fan-out** - for each NAICS->domain edge, join against existing NAICS<->ISIC / NAICS<->NACE crosswalks and emit parallel edges so European / UN users get the bridge too. Pure SQL.

### The four edge kinds

Derived purely from both endpoints' categories (`system_id.startswith('domain_')` -> category):

| `edge_kind`          | Source category | Target category | Example |
|----------------------|-----------------|-----------------|---------|
| `standard_standard`  | standard        | standard        | NAICS 6211 <-> ISIC 8620 |
| `standard_domain`    | standard        | domain          | NAICS 6212 <-> domain_dental (generated by this plan) |
| `domain_standard`    | domain          | standard        | reverse of above, when queried from the domain side |
| `domain_domain`      | domain          | domain          | domain_truck_freight <-> domain_truck_cargo (future) |

Pure function, no schema column, computed on read.

## Anchor table design

Shape:

```json
{
  "domain_ai_deployment":     { "naics": ["5415"] },
  "domain_ai_governance":     { "naics": ["5415", "5416"] },
  "domain_ai_data":           { "naics": ["5415", "518"] },
  "domain_insurance_product": { "naics": ["524"] },
  "domain_dental":            { "naics": ["6212"] },
  "domain_telehealth":        { "naics": ["6213", "5415"] },
  "domain_iso_9001":          { "naics": ["5416"], "note": "weak_anchor" }
}
```

Rules:

- **Anchor granularity:** prefer the most specific NAICS code where only one clearly applies (e.g., `6212` for dental); otherwise multiple 2-3 digit anchors.
- **Minimum one anchor per domain.** No zero-anchor rows. If a domain truly doesn't fit NAICS, anchor to the best sector match and flag `"note": "weak_anchor"`.
- **Max three anchors** to keep edge count sane.

## Edge generation rules

For each anchor row `(domain_system_id, [naics_code, ...])`:

1. Resolve domain level-1 codes: `SELECT code FROM classification_node WHERE system_id = $1 AND level = 1`.
2. For each NAICS anchor x each level-1 code, emit:
   ```
   (source_system='naics_2022', source_code=<naics>,
    target_system=<domain>, target_code=<level1>,
    match_type='broad',
    provenance='derived:sector_anchor:v1')
   ```
3. Deduplicate, `ON CONFLICT DO NOTHING`.

Expected edge volume: ~419 domains x ~1.8 anchors x ~3-5 level-1 codes ~= **3,000-4,000 new edges**. Small on top of the existing 321K.

## ISIC / NACE fan-out

After NAICS->domain edges exist, a single SQL pass produces the European / UN mirror:

```sql
INSERT INTO equivalence (source_system, source_code, target_system, target_code, match_type)
SELECT DISTINCT
    e1.target_system AS source_system,
    e1.target_code   AS source_code,
    e2.target_system AS target_system,
    e2.target_code   AS target_code,
    'broad'          AS match_type
FROM equivalence e1
JOIN equivalence e2
  ON e1.source_system = 'naics_2022'
 AND e1.source_code   = e2.source_code
 AND e2.source_system = 'naics_2022'
 AND e2.target_system LIKE 'domain_%'
 AND e1.target_system IN ('isic_rev4','nace_rev2')
ON CONFLICT DO NOTHING;
```

## Edge-kind labeling surface

**REST API** - three endpoints return `Equivalence` objects:

- `GET /api/v1/systems/{id}/nodes/{code}/equivalences` - `edge_kind` on every item
- `GET /api/v1/systems/{id}/nodes/{code}/translations` - same
- `GET /api/v1/equivalences/stats` - new `group_by=edge_kind` option alongside the system-pair matrix

New filter query params on the first two:

```
GET /api/v1/systems/naics_2022/nodes/6211/equivalences?edge_kind=standard_standard
GET /api/v1/systems/naics_2022/nodes/6211/equivalences?edge_kind=standard_domain,domain_standard
```

Default behavior unchanged: all edges returned, now labeled.

**MCP server** - tools that return equivalence items get the field:

- `get_equivalences` / `translate_code` / `translate_across_all_systems` - items carry `edge_kind`
- `get_crosswalk_coverage` - response carries the computed pair `edge_kind`
- `classify_business` - when bridging edges surface inline, they carry `edge_kind`

New tool:

- `list_crosswalks_by_kind(edge_kind, system_id?)` - counts + samples for the requested kind. Lets agents answer "show me every standard-to-domain bridge touching NAICS" in one call.

**Response schema** (Pydantic + TS):

```python
class EquivalenceEdge(BaseModel):
    source_system: str
    source_code: str
    target_system: str
    target_code: str
    match_type: Literal['exact', 'broad', 'narrower', 'partial', 'related']
    edge_kind: Literal['standard_standard', 'standard_domain', 'domain_standard', 'domain_domain']
    source_category: Literal['standard', 'domain']
    target_category: Literal['standard', 'domain']
```

`source_category` + `target_category` are redundant with `edge_kind` but cheap to include and let lazy consumers filter without parsing the label.

## What this plan does NOT cover

- **Match type refinement.** Everything goes in as `match_type='broad'`. Tightening to `narrower` / `exact` per sector is a follow-up.
- **Curated v2 anchors.** Every new edge carries `provenance='derived:sector_anchor:v1'`. Human-curated v2 is a separate plan.
- **Non-NAICS anchors.** Some domains (e.g., `domain_iso_9001`) may anchor better in CPC or ESCO. Phase 1 keeps everything NAICS-anchored.
- **Domain-to-domain edges.** `domain_domain` is defined in the edge-kind taxonomy but no such edges are generated by this plan; they're opt-in follow-up work.
- **Frontend UI changes beyond optional badges.** Existing two-category rendering handles bridged vs unbridged gracefully.
- **MCP tool signatures** for existing tools beyond adding the `edge_kind` field. `classify_business` / `translate_across_all_systems` already use the `equivalence` table; they pick up the new edges and labels automatically.
