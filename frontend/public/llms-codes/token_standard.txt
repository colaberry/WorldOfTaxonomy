[TS] Token Standards -- Token standards define a set of technical rules and interfaces that blockchain tokens must follow to ensure interoperability, consistency, and ease of integration across wallets, exchanges, and decentralized applications. They specify functions such as transfer, balance tracking, and metadata handling, enabling developers to create tokens that work reliably within a given blockchain ecosystem. Common examples include ERC-20 for fungible assets and ERC-721 for non-fungible assets.
[TS.01] ERC-20 (Fungible Token) -- TS.01 defines the ERC-20 standard, the most widely used specification for creating fungible tokens on the Ethereum blockchain. It mandates a set of functions such as totalSupply, balanceOf, transfer and allowance, enabling token interoperability across wallets, exchanges and smart contracts.
[TS.02] ERC-721 (NFT) -- TS.02 denotes the ERC-721 token standard, a protocol on Ethereum for creating non-fungible tokens (NFTs). It defines unique, indivisible assets with distinct metadata, enabling ownership, transfer, and verification of digital collectibles, artwork, and other singular items on the blockchain.
[TS.03] ERC-1155 (Multi-Token) -- ERC-1155 is a versatile token standard that enables a single smart contract to manage multiple token types, both fungible and non-fungible, within one ecosystem. It reduces transaction costs and gas fees by batching operations, allowing efficient batch transfers and balance queries for diverse assets in decentralized applications.
[TS.04] ERC-4626 (Tokenized Vault) -- ERC-4626 defines a standardized interface for tokenized vaults that issue shares representing a claim on underlying assets, enabling interoperable deposit, withdrawal and accounting across DeFi protocols. It builds on ERC-20 and ensures consistent yield bearing token behavior.
[TS.05] BEP-20 (BNB Chain) -- BEP-20 is the token standard used on the BNB Chain, defining a common set of rules for creating and managing fungible tokens. It mirrors Ethereum's ERC-20 interface, specifying functions such as transfer, balance inquiry, and allowance, enabling interoperability across wallets, decentralized applications, and exchanges within the Binance ecosystem.
[TS.06] SPL Token (Solana) -- The SPL token is Solana's native token standard, defining a set of on-chain rules for creating, transferring, and managing fungible and non-fungible assets. It leverages Solana's high-throughput architecture, enabling low-fee, fast transactions for decentralized finance, gaming, and other tokenized applications.
[TS.07] TRC-20 (Tron) -- TS.07 refers to TRC-20, the token standard on the Tron network that defines a uniform interface for creating and handling fungible tokens. It mirrors Ethereum's ERC-20 functions such as transfer, balance query and allowance, facilitating interoperability across wallets, exchanges and decentralized applications.
[TS.08] FA2 (Tezos) -- FA2 is Tezos' multi-asset token standard that supports fungible, non-fungible, and semi-fungible tokens within a single contract. It defines entrypoints for transferring tokens, querying balances, and managing operators, allowing flexible creation and control of diverse digital assets.
[TS.09] ASA (Algorand) -- ASA, or Algorand Standard Asset, is Algorand's native token framework that lets developers create fungible, non-fungible, or hybrid assets directly on the Algorand blockchain. The standard defines a set-of-parameters-including total supply, decimals, and manager addresses-to ensure interoperable, secure, and low-cost asset issuance and transfer.
[TS.10] CW-20 (CosmWasm) -- TS.10 defines the CW-20 token standard, a fungible-token specification for smart contracts built with CosmWasm on the Cosmos ecosystem. Modeled after Ethereum's ERC-20, it outlines a common interface for balance queries, transfers, allowances and metadata, enabling interoperable token creation and interaction across Cosmos-based blockchains.
[TS.11] Security Token (STO) -- TS.11 designates security tokens, blockchain-based digital assets that represent a regulated financial interest such as equity, debt, revenue share or dividend rights. They must comply with securities laws and typically require KYC/AML verification, offering investors legal protections and enforceable ownership claims, unlike utility tokens that merely grant access to a service.
[TS.12] Soulbound Token (SBT) -- TS.12 defines Soulbound Tokens as non-transferable digital assets that are permanently linked to a single wallet or identity, often used to represent credentials, achievements, or affiliations. Because they cannot be moved or traded, SBTs provide immutable proof of personal or institutional attributes without the risk of resale or speculation.
[TS.13] Wrapped Token -- A wrapped token is a blockchain-based representation of an asset that originates on a different network, created to allow that asset to be used on the host chain. It is typically backed one-to-one by the original token, with a smart contract managing deposits, minting, and redemption to maintain the peg.
[TS.14] Stablecoin Standard -- TS.14 sets out the technical and operational criteria for stablecoins, detailing acceptable collateral models, peg mechanisms, governance structures, and audit procedures to maintain a reliable value relative to a reference asset. It serves as a blueprint for developers and issuers seeking to create tokens that achieve price stability and meet regulatory expectations.
